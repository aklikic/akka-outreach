# Data Model: BDR LinkedIn Outreach Tool

**Feature**: 001-bdr-outreach | **Phase**: 1 | **Date**: 2026-03-25

## Account State

Account state lives entirely inside `OutreachGenerationWorkflow` — there is no separate entity. The `Account` domain record is used directly as the workflow's state type.

### Account (domain record — `com.example.domain.Account`)

| Field | Type | Notes |
|-------|------|-------|
| `accountId` | String | UUID, workflow ID |
| `companyName` | String | Required |
| `industry` | String | Required |
| `website` | String | Required |
| `targetPersona` | String | Required (e.g. "VP Engineering") |
| `notes` | String | Optional, may be empty |
| `status` | AccountStatus | NEW \| MESSAGE_READY \| RESPONDED \| ERROR |
| `signals` | List\<String\> | Populated after signal research; empty until then |
| `messages` | List\<OutreachMessage\> | 3 messages; empty until generation completes |
| `hubspotId` | String | Populated after HubSpot sync; null until then |
| `createdAt` | Instant | Set on creation |

### Status Transitions

```
NEW ──────────────────────────→ MESSAGE_READY ──→ RESPONDED
 │                                   │
 │  (generation fails after retry)   │  (re-generation allowed)
 └───────────────────────────────────→ ERROR ──→ NEW (on re-trigger)
```

## Workflows

### OutreachGenerationWorkflow

**Component ID**: `outreach-generation`
**Entity key**: `accountId`
**Package**: `com.example.application.OutreachGenerationWorkflow`

#### State
```
record State(
  String accountId,
  CompanyDetails details,      // companyName, industry, website, targetPersona
  List<String> signals,        // populated after step 1
  List<OutreachMessage> messages  // populated after step 2
)
```

#### Commands

| Method | Parameter | Action |
|--------|-----------|--------|
| `create` | `CreateRequest(accountId, companyName, industry, website, targetPersona, notes)` | Initialise workflow state with status=NEW |
| `startGeneration` | — | Transition to `researchSignalsStep` |
| `markAsResponded` | — | Update state status=RESPONDED |
| `notifyDone` | `String bulkJobId` | Called by `BulkGenerationWorkflow` on completion (success or error); no-op if `bulkJobId` absent |

#### Steps

| Step Method | Action | On Success | On Failure (after 1 retry) |
|-------------|--------|------------|---------------------------|
| `researchSignalsStep` | Call `SignalResearchAgent.research(details)` | store signals in state → `generateMessagesStep` | → `errorStep` |
| `generateMessagesStep` | Call `MessageGenerationAgent.generate(details, signals)` | store messages in state → `updateStatusStep` | → `errorStep` |
| `updateStatusStep` | Update workflow state: status=MESSAGE_READY, signals, messages | → thenEnd | → `errorStep` |
| `errorStep` | Update workflow state: status=ERROR; call `BulkGenerationWorkflow.notifyDone` if `bulkJobId` present | → thenEnd | — |

**Step timeout**: 60s for agent steps, 5s for update/error steps.

---

### BulkGenerationWorkflow

**Component ID**: `bulk-generation`
**Entity key**: `bulkJobId` (UUID generated by caller)
**Package**: `com.example.application.BulkGenerationWorkflow`

#### State
```
record State(
  List<String> accountIds,   // ordered list of accountIds to process
  int currentIndex           // index of the account currently being processed
)
```

#### Commands

| Method | Parameter | Action |
|--------|-----------|--------|
| `start` | `StartRequest(bulkJobId, accountIds)` | Initialise state, transition to `delegateStep` |
| `notifyDone` | `NotifyDoneRequest(accountId, boolean success)` | Advance `currentIndex`; transition to `delegateStep` or `thenEnd` |

#### Steps

| Step Method | Action | On Success | On Failure |
|-------------|--------|------------|-----------|
| `delegateStep` | Call `OutreachGenerationWorkflow.startGeneration(accountIds[currentIndex])` passing `bulkJobId` | → pause (AWAITING_COMPLETION); wait for `notifyDone` command | — |

**Note**: `BulkGenerationWorkflow` does not call agents directly. All generation logic lives in `OutreachGenerationWorkflow`. `BulkGenerationWorkflow` advances to the next account only after receiving the `notifyDone` push command.

## Agents

### SignalResearchAgent

**Component ID**: `signal-research`
**Role** (`@AgentRole`): "Company signal researcher. Given company details, returns exactly 3 intent signals describing recent activity at the company."
**Command handler**: `research(CompanyDetails details)` → `List<String>`
**@FunctionTool**: `getSignalsForCompany(String companyName)` — calls `SignalPool.signalsFor(companyName)`
**Model**: default (googleai-gemini)

### MessageGenerationAgent

**Component ID**: `message-generation`
**Role** (`@AgentRole`): "LinkedIn outreach message writer. Given company details, intent signals, and a target persona, generates exactly 3 LinkedIn messages: a connection request, a first follow-up, and a second follow-up."
**Command handler**: `generate(GenerateRequest request)` → `Messages`
**Input record**: `GenerateRequest(CompanyDetails details, List<String> signals)`
**Output record**: `Messages(List<OutreachMessage> messages)` — use `responseConformsTo(Messages.class)`
**Model**: default (googleai-gemini)

## Consumer

### HubSpotSyncConsumer

**Component ID**: `hubspot-sync`
**Source**: `@Consume.FromWorkflow(OutreachGenerationWorkflow.class)` — `onUpdate(Account state)`
**Handles**: Updates where `state.status() == RESPONDED` and `state.hubspotId() == null`
**Action**: Computes `hubspotId` via `HubSpotService.syncAccount(companyName)`, then calls `OutreachGenerationWorkflow.recordHubSpotSync(hubspotId)` via `ComponentClient`.
